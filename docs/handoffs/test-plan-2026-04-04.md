## Test Class Coverage (permuplate-tests)

### `PermuteTest` — Basic type permutation
**What it covers:** `@Permute` range, nested class/interface promotion, string constants, cross-product via `@PermuteVar`

Key scenarios to verify:
- From/to range: only classes in [from, to] are generated (not template class name)
- `className = "Join${i}"` — template `Join2` with from=3 generates Join3, Join4, ... (no Join2)
- Nested static class promoted to top-level in generated output
- `strings = {"name=Foo"}` — string constant available in expressions
- `extraVars = {@PermuteVar(varName="k", from=2, to=3)}` — cross-product: all (i,k) combinations

Regression check: ensure template class itself is NOT generated as one of the outputs.

---

### `PermuteDeclrTest` — Field and variable renaming
**What it covers:** `@PermuteDeclr` on fields, constructor params, for-each variables

Key scenarios:
- `@PermuteDeclr(type="Callable${i}", name="c${i}")` on a field → type AND name change, ALL usages in class body renamed
- `@PermuteDeclr` on constructor param → rename propagates only in constructor body (not class-wide)
- `@PermuteDeclr` on for-each variable → rename propagates only in loop body
- Two annotated fields in same class — first field's rename must not corrupt second
- Dual for-each loops — each loop variable scoped independently

**Critical regression:** Field transform must run BEFORE for-each transform (fields have class-wide scope; if for-each runs first, the field rename may not be visible to the loop body walker).

---

### `PermuteParamTest` — Sentinel parameter expansion
**What it covers:** `@PermuteParam` anchor expansion, fixed params before/after sentinel

Key scenarios:
- `@PermuteParam(varName="j", from="1", to="${i-1}", type="Object", name="o${j}")` — sentinel `o1` becomes `o1, o2, ..., o(i-1)`
- Fixed param before sentinel (e.g. `String ctx`) preserved in position
- Fixed param after sentinel (e.g. `List<Object> results`) preserved in position
- Anchor expansion: call sites where the sentinel name appears as an argument get expanded to the full sequence
- Multiple sentinels in same method: processed sequentially (while loop, each pass removes one sentinel)

**Critical regression:** Anchor at call sites must use indexOf on the original param list to correctly position the expansion relative to before/after params.

---

### `ExpressionFunctionsTest` — Built-in JEXL functions
**What it covers:** `alpha()`, `lower()`, `typeArgList()`

Key scenarios:
- `alpha(1)` → `"A"`, `alpha(26)` → `"Z"`, `alpha(0)` → throws, `alpha(27)` → throws
- `lower(1)` → `"a"`, `lower(26)` → `"z"`
- `typeArgList(1, 3, "T")` → `"T1, T2, T3"`
- `typeArgList(1, 3, "alpha")` → `"A, B, C"`
- `typeArgList(1, 3, "lower")` → `"a, b, c"`
- `typeArgList(5, 3, "T")` → `""` (empty range — from > to returns empty string, not error)
- End-to-end: `@Permute` with `className="Step${alpha(i)}"` generates StepA, StepB, StepC

**Critical regression:** Empty range `typeArgList(from, to, style)` when `from > to` MUST return `""` — not throw. This is used by `Join6Second.fn()` to build the type arg list for zero fact params.

### `PermuteTypeParamTest` — `@PermuteTypeParam`
**What it covers:** Explicit and implicit type parameter expansion, bounds propagation, alpha naming, validation rules

Key scenarios:
1. **Explicit, no bounds** — `@PermuteTypeParam(varName="j", from="1", to="${i}", name="T${j}")` → `Condition3<T1, T2, T3>`
2. **Explicit, with bounds** — `T1 extends Comparable<T1>` → each generated param has matching bounds `T2 extends Comparable<T2>`, etc.
3. **Explicit range** — from=2, to=4 generates RangeCondition2, RangeCondition3, RangeCondition4
4. **Implicit (via @PermuteParam type="T${j}")** — no `@PermuteTypeParam` needed; class type params auto-expand when `@PermuteParam` references them
5. **Implicit with bounds** — bounds propagate same as explicit
6. **Fixed type param survives** — `<T1, R>` where only T1 expands: `Transformer3<T1, T2, T3, R>` — R stays
7. **Alpha naming** — `name="${alpha(j)}"` → `AlphaCondition3<A, B, C>`
8. **Alpha with bounds** — `A extends Comparable<A>` → `B extends Comparable<B>`, `C extends Comparable<C>`
9. **R1 validation** — return type references expanding type param → compile error
10. **R3 validation** — name prefix mismatch → compile error
11. **R4 validation** — `from > to` → compile error ("invalid range")

**Critical regression:** After adding G3 alpha fix, verify that PermuteTypeParamTransformer still runs before applyExtendsExpansion in InlineGenerator.generate() (line ordering matters).

---

### `PermuteReturnTest` — `@PermuteReturn`
**What it covers:** APT explicit mode, inline implicit inference, boundary omission, `when="true"`, validation

Key scenarios:
1. **APT explicit** — `Node1..Node4` chain: each `next()` returns `Node${i+1}`; Node4.next() omitted (Node5 not in generated set)
2. **APT explicit when="true"** — ForcedChain2.next() generates even though ForcedChain3 not in generated set
3. **Inline implicit basic boundary omission** — `Step1<T1>` with `join()` returning `Step2<T1,T2>`: Step4.join() omitted (Step5 not in generated set)
4. **Inline implicit parameter type inferred** — `Source<T2>` in join() parameter: T2 becomes T3 at i=2
5. **Inline implicit fixed type param survives** — R passes through in `Step2<T1,T2,R>` → `Step3<T1,T2,T3,R>`
6. **Alpha naming explicit** — `AlphaStep2<A,B>` through `AlphaStep4<A,B,C,D>`; `next()` returns raw `AlphaStep${i+1}` (no typeArgs since next-arity params out of scope)
7. **V2** — typeArgVarName without typeArgTo → compile error
8. **V3** — typeArgFrom > typeArgTo → compile error "invalid"
9. **V6** — typeArgs AND typeArgVarName both set → compile error

**Critical regression:** Boundary omission fires correctly — if the evaluated `className` at the current arity is NOT in the generated set, the method is silently omitted. `when="true"` overrides. Verify Join6First has no join() method in the Drools example.

### `PermuteMethodTest` — `@PermuteMethod` + extends expansion
**What it covers:** G3 extends clause expansion (4 tests), APT explicit @PermuteMethod

Key scenarios:

**Extends expansion — same-N formula (critical regression target):**
1. `testExtendsClauseImplicitExpansion` — `Join1First<T1> extends Join1Second<T1>` with T+number expansion
   - forI=1: `extends Join1Second<T1>` (NOT Join2Second — forward reference was the old bug)
   - forI=2: `extends Join2Second<T1, T2>` (NOT Join3Second)
   - forI=3: `extends Join3Second<T1, T2, T3>`
   - `doesNotContain("T2")` at forI=1, `doesNotContain("T3")` at forI=2

2. `testExtendsClauseUnchangedWhenNotInGeneratedSet` — `Step1<T1> extends BaseStep<T1>`
   - BaseStep has different prefix → NOT expanded → stays as `extends BaseStep<T1>`

3. `testExtendsClauseAlphaNaming` — `Join0First<DS, @PermuteTypeParam(...) A> extends Join0Second<DS, A>`
   - forI=1: `Join1First<DS, A>`, `extends Join1Second<DS, A>`, doesNotContain("B")
   - forI=2: `Join2First<DS, A, B>`, `extends Join2Second<DS, A, B>`
   - forI=3: `Join3First<DS, A, B, C>`, `extends Join3Second<DS, A, B, C>`

4. `testExtendsClauseAlphaNamingNoFixedPrefix` — `Step0First<@PermuteTypeParam(...) A> extends Step0Second<A>` (no DS)
   - forI=1: `Step1First<A>`, `extends Step1Second<A>`, doesNotContain("B")
   - forI=2: `Step2First<A, B>`, `extends Step2Second<A, B>`
   - Verifies the `size <= size` boundary in isPrefix (full match at i=1, strict prefix at i=2)

5. `testAptExplicitPermuteMethod` — APT mode with `@PermuteMethod` generating multiple overloads

**Critical regression:** Both doesNotContain assertions at forI=1 in tests 1, 3, 4 are the specific forward-reference regression guard. If the `+1` formula sneaks back in, these fail first.

### `DegenerateInputTest` — Error paths and validation
**What it covers:** All `@Permute` attribute error conditions with message content and source-position assertions

Key scenarios:
- Missing `varName`, missing `from`, missing `to` — appropriate error messages
- `from > to` — invalid range error
- `className` contains no `${}` — R1 error (must have variable in className)
- `className` prefix doesn't match template class name prefix — error
- All errors must have at least element-level precision (no bare `messager.printMessage(ERROR, msg)`)

**Critical regression:** Every new error path added to the processor must appear in DegenerateInputTest.

---

### `PrefixValidationTest` — Annotation string validation (R2/R3/R4)
**What it covers:** String literal substring matching (R2), orphan variable detection (R3), no-anchor detection (R4)

Key scenarios:
- R2: every static literal in the annotation string must appear as a substring of the target name
- R3: variables that appear before the first static literal are "orphan" (no anchoring prefix) — error if the name has no prefix
- R4: annotation string with no static literal at all — error
- R2 short-circuits R3/R4: if R2 fails (literal not found), skip orphan/anchor checks (undefined)
- Adjacent variables are collective: `${v1}${v2}Callable${v3}` — v1 and v2 collectively cover the prefix region

## Key Regression Scenarios

These are the scenarios most likely to break after changes. Run targeted tests for these before committing:

### After any change to `InlineGenerator.applyExtendsExpansion()`:
```bash
/opt/homebrew/bin/mvn test -pl permuplate-tests -Dtest=PermuteMethodTest
```
Check specifically:
- `testExtendsClauseImplicitExpansion`: forI=1 does NOT have T2, forI=2 does NOT have T3
- `testExtendsClauseAlphaNaming`: all three arity levels have correct DS+alpha args in extends
- `testExtendsClauseUnchangedWhenNotInGeneratedSet`: BaseStep NOT expanded

### After any change to `InlineGenerator.generate()` (pipeline ordering):
```bash
/opt/homebrew/bin/mvn test -pl permuplate-tests -Dtest=PermuteTypeParamTest,PermuteMethodTest
```
The pipeline order inside `generate()` matters:
1. `PermuteTypeParamTransformer.transform()` (G1) — MUST run before applyExtendsExpansion
2. `applyPermuteMethod()` (G3) — MUST run before PermuteDeclrTransformer
3. `PermuteDeclrTransformer.transform()`, `PermuteParamTransformer.transform()`
4. `applyPermuteReturn()`, `applyImplicitInference()`
5. `applyExtendsExpansion()` — MUST run after G1 (needs postG1TypeParams)

### After any change to `PermuteTypeParamTransformer`:
```bash
/opt/homebrew/bin/mvn test -pl permuplate-tests -Dtest=PermuteTypeParamTest,PermuteMethodTest
# Also test the Drools example since Consumer/Predicate use @PermuteTypeParam
/opt/homebrew/bin/mvn test -pl permuplate-mvn-examples -am
```

### After any change to `EvaluationContext` or JEXL functions:
```bash
/opt/homebrew/bin/mvn test -pl permuplate-tests -Dtest=ExpressionFunctionsTest,PermuteMethodTest
```
Specifically: `typeArgList(1, 0, "alpha")` must return `""` not throw.

### After any change to `RuleDefinition.wrapPredicate`:
```bash
/opt/homebrew/bin/mvn test -pl permuplate-mvn-examples -am -Dtest=RuleBuilderTest#testArity3IntermediateFilter,RuleBuilderTest#testArity2MultipleFilters
```
The arity-truncation logic is tested by multi-arity chains with intermediate filters.

### After any change to boundary omission logic:
```bash
/opt/homebrew/bin/mvn test -pl permuplate-tests -Dtest=PermuteReturnTest
/opt/homebrew/bin/mvn test -pl permuplate-mvn-examples -am -Dtest=RuleBuilderTest#testArity6LeafNodeCompiles
```
`testArity6LeafNodeCompiles` is a compile-time test — if Join6First incorrectly has a join() method, this test won't even compile.

## Things That Are NOT Tested (gaps to be aware of)

1. **IDE plugin validation** — `permuplate-ide-support`'s `AnnotationStringAlgorithm` is tested via `PrefixValidationTest` but no actual IntelliJ plugin exists yet
2. **Cross-module APT dependencies** — the workaround (`strings={"max=N"}`) is demonstrated but not exhaustively tested
3. **Large arity ranges** — tests go up to i=10 (DogFoodingTest) but very large ranges (50+) untested for performance
4. **Template with multiple `@Permute` annotations on the same class** — currently each class can only have one `@Permute`
5. **Drools Phase 2 (First/Second split)** — not yet implemented. This is the next major feature.
6. **`join()` typed parameters** — currently `Function<DS, DataSource<?>>` (wildcard). Typed `Function<DS, DataSource<B>>` is impossible with current Permuplate capabilities (B not in scope).
