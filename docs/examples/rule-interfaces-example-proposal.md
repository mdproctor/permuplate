## Background

Analysis of a [Drools rule DSL proof-of-concept](https://github.com/mdproctor/drools/blob/droolsvol2/droolsvol2/src/test/java/org/drools/core/RuleBuilderTest.java) revealed a real-world use case where Permuplate's inline generation (Maven plugin) would eliminate a large family of hand-written boilerplate. The DSL uses two core families of functional interfaces — predicates and consumers — each needing variants for 1 through N facts.

---

## What This Example Demonstrates

This example shows Permuplate's Maven plugin (`inline = true`) generating **two independent nested interface families** from a single parent class. It is the most realistic example in the project: the pattern arises naturally in any rule engine, event processor, or fluent DSL where actions and conditions must handle variable numbers of facts.

**The generated output** is a class containing `Condition1`–`Condition5` and `Action1`–`Action5` as nested interfaces — ready to use as lambda targets without any code generation visible to the user.

## Template Design

```java
// src/main/permuplate/io/quarkiverse/permuplate/example/RuleInterfaces.java

package io.quarkiverse.permuplate.example;

import io.quarkiverse.permuplate.Permute;
import io.quarkiverse.permuplate.PermuteParam;

/**
 * Template that generates two families of functional interfaces for use in
 * rule-engine-style DSLs: Condition1–Condition5 (boolean predicate) and
 * Action1–Action5 (void consumer). All variants are nested inside
 * {@code RuleInterfaces} for a clean, self-contained API.
 *
 * <p>Generated by the Permuplate Maven plugin (inline = true). The template
 * lives in {@code src/main/permuplate/} and is never compiled directly.
 *
 * <p>Usage example:
 * <pre>{@code
 * RuleInterfaces.Condition2 filter = (person, account) -> person.age() > 18;
 * RuleInterfaces.Action2    action = (person, account) -> account.approve();
 * }</pre>
 */
public class RuleInterfaces {

    /**
     * Template for a boolean predicate over N facts.
     * Generates {@code Condition1} through {@code Condition5}.
     * {@code keepTemplate = true} retains {@code Condition1} (a 1-fact predicate
     * is genuinely useful, not merely a scaffold).
     */
    @Permute(varName = "i", from = 2, to = 5,
             className = "Condition${i}",
             inline = true,
             keepTemplate = true)
    public interface Condition1 {
        /**
         * Evaluates this condition against the given facts.
         *
         * @return {@code true} if the condition is satisfied
         */
        boolean test(
                @PermuteParam(varName = "j", from = "1", to = "${i}",
                              type = "Object", name = "fact${j}") Object fact1);
    }

    /**
     * Template for a void consumer over N facts.
     * Generates {@code Action1} through {@code Action5}.
     * {@code keepTemplate = true} retains {@code Action1}.
     */
    @Permute(varName = "i", from = 2, to = 5,
             className = "Action${i}",
             inline = true,
             keepTemplate = true)
    public interface Action1 {
        /**
         * Executes this action with the given facts.
         */
        void execute(
                @PermuteParam(varName = "j", from = "1", to = "${i}",
                              type = "Object", name = "fact${j}") Object fact1);
    }
}
```

## What This Example Shows (and Does Not Show)

### Shows

- `inline = true` with the Maven plugin generating nested interfaces
- `keepTemplate = true` retaining the 1-fact variants (genuinely useful)
- **Two independent `@Permute` templates in the same parent class**
- `@PermuteParam` expanding a sentinel parameter into N facts on an interface method (not just a class method — verifies interface support)
- The complete `src/main/permuplate/` template directory convention
- Clean, lambda-friendly API: `Condition3 c = (a, b, c) -> a.age() > b.age()`

### Does Not Show (by design — out of scope)

- Generic type parameter arity (`<T1, T2, T3>` — see gap analysis)
- Stateful return type narrowing per join step
- Type-safe fact binding (everything is `Object`)
