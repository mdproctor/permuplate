## Item 2 — `@PermuteFilter("i > 1")` replaces `max(2, i)` on filterLatest

### Problem
The single-fact filter suppression at arity 1 uses a `max()` trick that encodes intent in a
from= arithmetic expression:
```java
@PermuteMethod(varName = "x", from = "${max(2, i)}", to = "${i}", name = "filter")
```
At i=1: from=2, to=1 → empty range → method suppressed. The suppression is implicit.

### Fix
Use `@PermuteFilter` on the method to express suppression explicitly:
```java
@PermuteFilter("i > 1")
@PermuteMethod(varName = "x", from = "${i}", to = "${i}", name = "filter")
```
At i=1: filter condition false → method removed entirely before `@PermuteMethod` runs.
At i≥2: filter passes → `@PermuteMethod` generates one clone `x=i`.

Reads: "suppress at arity 1; at arity 2+ generate one overload."

### Permuplate changes
None — `@PermuteFilter` on methods already ships.

### Tests
- `filter(Predicate2<DS,A>)` absent on `Join1First` (i=1 has all-facts filter only)
- `filter(Predicate2<DS,B>)` present on `Join2First`, `filter(Predicate2<DS,C>)` on `Join3First` etc.
- Existing `RuleBuilderTest` single-fact filter tests pass

### Problem
Join0First has five fluent methods all returning `this`, each annotated with `@PermuteSelf`:
```

filterLatest (via @PermuteMethod) — @PermuteSelf (line 307)
```

```java

                      typeArgs="'<END, DS, ' + alphaList + '>'")
public static non-sealed class Join0First<...> {
```

`@PermuteDefaultReturn` fires on all Object-returning methods without explicit `@PermuteReturn`.

## Item 4 — Consumer1 + Predicate1 → single cross-product template

### Problem
`Consumer1.java` (24 lines) and `Predicate1.java` (23 lines) are structurally identical:
same `@Permute` range, same `@PermuteTypeParam`, same `@PermuteParam` expansion.
Only the class name prefix, method name, and return type differ:

| | Consumer | Predicate |
|---|---|---|
| className prefix | `Consumer` | `Predicate` |
| method name | `accept` | `test` |
| return type | `void` | `boolean` |

### Fix
Single cross-product template using `@PermuteVar` (already ships):

```java
// Single file replaces both Consumer1.java and Predicate1.java
@Permute(varName = "i", from = "2", to = "7", className = "${F}${i}",
         inline = false, keepTemplate = false,
         macros = {"method=${F == 'Consumer' ? 'accept' : 'test'}",
                   "ret=${F == 'Consumer' ? 'void' : 'boolean'}"})
@PermuteVar(varName = "F", values = {"Consumer", "Predicate"})
public interface FunctionalTemplate1<A> {

    @PermuteTypeParam(varName = "j", from = "1", to = "${i-1}", name = "${alpha(j)}")
    @PermuteDeclr(type = "${ret}", name = "${method}")
    @PermuteParam(varName = "j", from = "1", to = "${i-1}",
                  type = "${alpha(j)}", name = "${lower(j)}")
    void accept(A a);
}
```

Cross-product: `i∈2..7` × `F∈{Consumer,Predicate}` → 12 interfaces (Consumer2..7, Predicate2..7).

**Arity-1 handling**: Consumer1 and Predicate1 (the arity-1 interfaces) are kept as small
hand-written files — they have one type parameter and the special DS-only signature that doesn't
follow the parameterized pattern cleanly. Two 8-line files replace the two 24/23-line templates.

**Alternative**: Use `from="1"` and `className="${F}${i+1}"` to also generate Consumer2..7 from
the cross-product. The template itself (`FunctionalTemplate1`) has arity 1 signature. With
`keepTemplate=false` the template class is not kept. Consumer1 and Predicate1 become explicit
hand-written interfaces (6-8 lines each) — total net reduction still ~20 lines.

**Chosen approach**: `from="2"`, `keepTemplate=false`, keep Consumer1.java and Predicate1.java
as minimal hand-written arity-1 interfaces (not templates). Avoids `${F}${i+1}` off-by-one
complexity.

### Permuplate changes
None — `@PermuteVar` + `@PermuteDeclr` on methods already ship.

### Tests
- All 12 generated interfaces exist as named types
- Consumer2..7 have `void accept(DS ctx, A a, B b, ...)` signatures
- Predicate2..7 have `boolean test(DS ctx, A a, B b, ...)` signatures
- Existing DSL tests using `Consumer4`, `Predicate3` etc. pass unchanged

## Item 7 — New annotation: `@PermuteBodyFragment`

### Problem
`@PermuteBody` strings can be long and cannot be shared across annotations on the same method.
In `RuleOOPathBuilder.java`, both conditional bodies share identical `steps.add(...)` code:

```java
@PermuteBody(when="${i==2}", body="{ steps.add(new OOPathStep(...)); rd.addOOPathPipeline(...); return end; }")
@PermuteBody(when="${i>2}", body="{ steps.add(new OOPathStep(...)); return new Path${i-1}<>(...); }")
//                                  ^^^^^^^^^^^^^^^^^^^^^^^^^ identical in both
```

Additionally, the escaped `\"` inside `@PermuteBody` strings for string literals is awkward
and fragile.

### Fix: `@PermuteBodyFragment` annotation

```java
@Target(ElementType.TYPE)
@Retention(RetentionPolicy.SOURCE)
public @interface PermuteBodyFragment {
    String name();       // reference key
    String value();      // the code fragment (JEXL-evaluated before insertion)
}
@Repeatable via @PermuteBodyFragments container
```

Usage on the enclosing class (or template class itself):
```java
@PermuteBodyFragment(name="addStep",
    value="steps.add(new OOPathStep(" +
          "(ctx, fact) -> (Iterable<?>) fn2.apply((PathContext<T>) ctx, (A) fact), " +
          "(ctx, child) -> flt2.test((PathContext<T>) ctx, (B) child)));")
public static class Path1<...> {
    @PermuteBody(when="${i==2}", body="${addStep} rd.addOOPathPipeline(rootIndex, steps); return end;")
    @PermuteBody(when="${i>2}", body="${addStep} return new @PermuteDeclr(type=\"...\") Path2<>(...);")
    public Object path(...) { ... }
}
```

`${addStep}` in the body string is substituted with the fragment value (which is itself
JEXL-evaluated first) before the body is parsed. Fragments are class-scoped (on template class
or enclosing class) and available to all `@PermuteBody` values within that scope.

### Implementation
- New `@PermuteBodyFragment` + `@PermuteBodyFragments` container in `permuplate-annotations`
- `InlineGenerator`: before `PermuteBodyTransformer` runs, collect `@PermuteBodyFragment`
  annotations from the template class and enclosing types. Build a fragment map
  `Map<String, String>`. Pass to `PermuteBodyTransformer` (or resolve in the body before
  transformer processes it) so `${fragmentName}` substitutions expand before JEXL evaluation.
- Stripping: `@PermuteBodyFragment`/`@PermuteBodyFragments` stripped from generated output.
- APT note: Maven plugin only; APT path emits NOTE if detected.

### Tests
- Unit test: fragment defined on class, used in `@PermuteBody` — fragment expands correctly
- Integration test: RuleOOPathBuilder compiles without the step duplication
- Test: fragment with JEXL expression inside value (`${i}` substituted correctly)
- Test: fragment defined on enclosing outer class — available to inner template class
- Existing OOPath tests pass

## Item 8 — New inference: `@PermuteParam` call-site expansion in method bodies

### Problem
`@PermuteParam` expands lambda parameter call sites (via `expandAnchorInStatement`) but does
NOT expand method body call sites. Hence `filterVar` needs an explicit `@PermuteBody`:

```java
@PermuteBody(body="{ rd.addVariableFilter(${typeArgList(1, m, 'v')}, predicate); return this; }")
```

The processor already knows all expanded parameter names (`v1..vm`). It just doesn't apply
call-site expansion to method body statements.

### Fix: body call-site anchor expansion

After `@PermuteParam` expands the method signature (adding `v2`, `v3`...), scan the method
body for call-expression arguments that contain the sentinel parameter name (`v1`). Expand
those argument lists by replacing `v1` with `v1, v2, ..., vm` (all expanded params in order,
stopping before the first non-expanded parameter).

Trigger conditions:
1. Method has `@PermuteParam` (not a lambda `@PermuteParam`)
2. Method body contains a call where the sentinel parameter appears as an argument
3. No `@PermuteBody` is present on the method (explicit wins)

### DSL impact
`filterVar` loses its `@PermuteBody`:
```java
@PermuteMethod(varName="m", from="2", to="3", name="filter")
@PermuteSelf  // (or removed if item 3 lands — @PermuteDefaultReturn covers it)
public <@PermuteTypeParam(...) V1> Object filterVar(
        @PermuteParam(varName="k", from="1", to="${m}", type="Variable<V${k}>", name="v${k}") Variable<V1> v1,
        @PermuteDeclr(type="Predicate${m+1}<DS, ${typeArgList(1, m, 'V')}>") Object predicate) {
    rd.addVariableFilter(v1, predicate);  // expanded to (v1, v2, predicate) or (v1, v2, v3, predicate)
    return this;
}
```

### Implementation
In `PermuteParamTransformer` (or `InlineGenerator` after `PermuteParamTransformer` runs):
after expanding the parameter list, collect the names of all expanded parameters (`v1..vm`).
Walk the method body looking for `MethodCallExpr` nodes where the argument list contains the
sentinel name (`v1`). Replace that argument with all expanded names in order (except the last
argument which comes after the sentinel in the template — preserve everything after the anchor).

This is a targeted expansion: only the argument position where the sentinel appears is
expanded. All other arguments are preserved.

### Tests
- Unit test: `@PermuteParam` on method → body call site expands correctly for m=2 and m=3
- Unit test: explicit `@PermuteBody` wins (inference does not fire when body annotation present)
- Unit test: multiple call sites with the anchor — all expand
- Integration test: `filterVar` compiles and works correctly without `@PermuteBody`
- Existing variable filter tests in `RuleBuilderTest` pass

## Item 9 — New inference: `@PermuteReturn` from `@PermuteBody` return expression

### Problem
When `@PermuteBody` is present AND contains a clear `return new X<>()` expression where
`X` is a generated class, the `@PermuteReturn` annotation is redundant — both express the
same class name.

In `scopeTemplate`:
```java
@PermuteReturn(className="${Scope}Scope", typeArgs="...", alwaysEmit=true)  // redundant
@PermuteBody(body="{ ... return new ${Scope}Scope<>(this, scopeRd); }")
```

The return class name `${Scope}Scope` is derivable from the body string via pattern matching.

### Inference rule
After `@PermuteBody` is applied and `@PermuteReturn` is absent: inspect the last statement of
the resulting method body. If it is `return new ClassName<>(...);` or
`return cast(new ClassName<>(...));` where `ClassName` matches a name in the generated class
set, infer `@PermuteReturn(className=ClassName, alwaysEmit=true)`.

**Type args inference**: For the simple case where the generated class exists in `allGeneratedNames`
and has the same type parameter count as the current class: infer `typeArgs` from
`classDecl.getTypeParameters()`. For complex cases (different arity, mixed args), fall back to
`Object` return (no inference).

### DSL impact
`scopeTemplate` drops `@PermuteReturn`:
```java
@PermuteMethod(varName="scope", values={"not","exists"}, name="${scope}",
               macros={"Scope=capitalize(scope)"})
// No @PermuteReturn needed — inferred from "return new ${Scope}Scope<>(...)"
@PermuteBody(body="{ ... return new ${Scope}Scope<>(this, scopeRd); }")
public Object scopeTemplate() { return null; }
```

However: the scope classes (`NotScope`, `ExistsScope`) are NOT in the generated class set of
`JoinBuilder` — they are in separate files. So `allGeneratedNames` for the JoinBuilder pass
won't contain them. This means the inference would NOT fire for scopeTemplate (correct behavior —
boundary omission would remove the method if it can't find the class in the set).

The `alwaysEmit=true` on the original `@PermuteReturn` prevents boundary omission. Without it,
the method would be omitted. So **for the scope case, explicit `@PermuteReturn(alwaysEmit=true)`
remains necessary** — the inference can't handle cross-file class references.

**Revised scope**: The inference fires only for classes within the same template's generated set.
The most applicable current DSL case would be if `joinBilinear` or `join` returned a class
within the generated set (they do, but already have explicit `@PermuteReturn`). **Net DSL
reduction: 0 for this specific case.** The inference is a general capability that benefits
other user templates more than the current DSL.

**Decision**: Still worth implementing as a general inference rule (makes Permuplate easier to
use), but the DSL won't immediately show reduction from it. Update the spec to reflect this
honestly.

### Tests
- Unit test: `@PermuteBody` with `return new GeneratedClass<>()` → return type inferred
- Unit test: cross-file return (class not in generated set) → inference does NOT fire; method
  needs explicit `@PermuteReturn(alwaysEmit=true)` or is omitted
- Unit test: multiple `@PermuteBody(when=)` on same method → inference uses last-statement
  of the first-matched body
- Existing tests all pass

## Item 10 — New special value: `@PermuteDefaultReturn(className="self")`

### Problem
`@PermuteDefaultReturn` requires verbose explicit expressions:
```java
@PermuteDefaultReturn(className="Join${i}First", typeArgs="'<END, DS, ' + alphaList + '>'")
```

The most common use case is "return the current generated class with its type parameters" —
which is exactly what `@PermuteSelf` does per-method, but there's no class-level equivalent
that uses self-inference.

### Fix: `className="self"` as reserved token

When `className="self"`:
- `className` → `classDecl.getNameAsString()` (already renamed for this permutation)
- `typeArgs` → derived from `classDecl.getTypeParameters()` (full list, same logic as `@PermuteSelf`)
- `typeArgs` attribute is ignored when `className="self"` (auto-computed)

```java
@PermuteDefaultReturn(className="self")  // all Object-returning methods → current class
public static non-sealed class Join0First<...> {
    // No @PermuteSelf or verbose typeArgs needed
}
```

### Implementation
In `applyInlineDefaultReturn` (InlineGenerator): after evaluating `className`, check if the
evaluated value equals `"self"` (literal string, not a JEXL expression). If so, use
`classDecl.getNameAsString()` as className and build typeArgs from
`classDecl.getTypeParameters()` via the same helper as `PermuteSelfTransformer.transform()`.
`"self"` is not evaluated as JEXL — it's a reserved literal checked before evaluation.

Similarly for `applyDefaultReturn` in `PermuteProcessor` (APT path).

### DSL impact
Join0First becomes:
```java
@Permute(varName="i", from="1", to="6", className="Join${i}First", ...)
@PermuteDefaultReturn(className="self")
```
Instead of:
```java
@PermuteDefaultReturn(className="Join${i}First", typeArgs="'<END, DS, ' + alphaList + '>'")
```

This is the cleanest form when combined with item 3 — one annotation, zero verbosity.

### Tests
- Unit test: `@PermuteDefaultReturn(className="self")` on a class → all Object-returning methods
  get the current generated class as return type with full type param list
- Integration test: Join0First with `className="self"` → filter(), index(), var() all return
  the correct typed class
- Test: `className="self"` with no type parameters → return type is bare class name
- Existing tests all pass
