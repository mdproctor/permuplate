Type-Safe Arity for Java

One Template. Infinite Arities.

Scala ships Tuple22, Function22, and the entire functional hierarchy to go with them. Java ships BiFunction — then stops.

The gap isn't just missing tuple types. It's every Callable, Function, and builder DSL that should exist for each arity, but doesn't. Any API that grows with the number of typed inputs means maintaining N near-identical classes by hand.

Permuplate closes that gap. One compilable template — your entire arity-varying infrastructure. Tuples, functional types, stateful builder DSLs. Type-safe. IDE-refactorable. Zero committed generated files.

The Problem

Java's Arity Ceiling

Java's Function goes to 2 arguments. Its Supplier is unary. Libraries like Vavr (Tuple1–8) and jOOL patch corners of the gap — but any API that varies by arity still forces N hand-maintained classes. That's the ceiling.

01

The Ceiling

Java's built-in functional types stop at 2. No native tuples. No Callable3, no Tuple5, no typed join builder. Hit arity 3 and you're writing these by hand:

All Identical Except

The number. That's it. Every declaration. Every parameter. Every call site.

- class Join2<A, B> + class Join3<A, B, C>
- Callable2<A, B> c2 + Callable3<A, B, C> c3
- DataSource<Tuple2<A,B>> right + DataSource<Tuple3<A,B,C>> right
03

Every Change

Fix a bug in Join3? Fix it in Join4, Join5... Join10. Every time. The tax compounds with every arity added.

Permuplate is the fourth option — the one that doesn't hurt.

The Killer Example

One Plate, Eight Prints

The template is a real, compilable Java class. Your IDE can navigate it, refactor it, and type-check it. At compile time, javac generates eight complete, type-safe classes. Type parameters grow in lockstep across the class, its Callable, and its DataSource<Tuple>: <A, B> → <A, B, C> → <A, B, C, D, E, F, G, H, I, J>. No Object anywhere.

1 → 8
The Template Join2.java 
@Permute(varName = "i", from = "3", to = "10",
         className = "Join${i}")
public class Join2<A,
        @PermuteTypeParam(varName = "k",
            from = "2", to = "${i}",
            name = "${alpha(k)}") B> {

    private @PermuteDeclr(
        type = "Callable${i}<${typeArgList(1,i,\"alpha\")}>",
        name = "c${i}") Callable2<A, B> c2;

    private @PermuteDeclr(
        type = "DataSource<Tuple${i}<${typeArgList(1,i,\"alpha\")}>>")
        DataSource<Tuple2<A, B>> right;
}

The Scope

One Pattern. Every Arity.

Permuplate isn't a tuple library — it's the tool that lets you build your own. Any type whose signature grows with the number of inputs is a template waiting to be written.

01

Tuple Families

Tuple2<A,B> through TupleN<A,…,N>. Type-safe multi-value containers with full IDE navigation. One record template. The whole family generated at compile time.

02

Functional Infrastructure

Callable1 through CallableN. Function, Consumer, Supplier — every variant Java stops short of. All type-checked, all refactorable, all from one template each.

03

Complete DSLs

Stateful builder chains where each step adds a type parameter and the compiler enforces arity. Rule engines. Query builders. Domain APIs. What would take thousands of lines of boilerplate is now one template file.

Get Started

Clone & Build

Permuplate is not yet on Maven Central — a public release is coming soon. For now, clone and install locally.

01

Clone and install

git clone https://github.com/mdproctor/permuplate.git
cd permuplate
mvn install -DskipTests
02

Add the processor to your Maven build

<plugin>
  <groupId>org.apache.maven.plugins</groupId>
  <artifactId>maven-compiler-plugin</artifactId>
  <configuration>
    <annotationProcessorPaths>
      <path>
        <groupId>io.quarkiverse.permuplate</groupId>
        <artifactId>permuplate-processor</artifactId>
        <version>1.0.0-SNAPSHOT</version>
      </path>
    </annotationProcessorPaths>
  </configuration>
</plugin>
03

Add the annotations dependency

<dependency>
  <groupId>io.quarkiverse.permuplate</groupId>
  <artifactId>permuplate-annotations</artifactId>
  <version>1.0.0-SNAPSHOT</version>
  <scope>provided</scope>
</dependency>

The Differentiator

Still Valid Java

Every other generator — Freemarker templates, Python scripts, Groovy code-gen — uses a template that is not valid Java. Your IDE can't navigate it, refactor it, or type-check it. When you rename a field, the generated classes break silently. Permuplate templates ARE valid, compilable Java. The same tools you already trust — the compiler, the IDE, Find Usages, Rename — all work, because the template is just Java.

01

IDE Native

Full IntelliJ and VS Code support. Navigate the template like any Java file. Set breakpoints. Use Find Usages. Rename a field and Permuplate propagates the change through every generated arity automatically.

02

Zero Runtime

Pure compile-time generation via standard javac. No runtime dependencies. No classpath pollution. Generated classes live in target/ — never committed to source control.

03

JEXL Expressions

Fully programmable loop bounds. from="${min}", to="${max * 2}". Inject from system properties at build time. Named string constants, built-in functions alpha(), lower(), cross-product generation — all in the annotation.

The Toolbox

Thirteen Tools. One Goal.

Annotation
Placed on
What it does
@Permute
class / method
Declares the permutation loop. The master switch — drives the outer i iteration from from to to, generating one class per value.
@PermuteVar
nested on @Permute
Adds extra integer loop axes for cross-product generation. BiCallable${i}x${k} produces 9 classes from one template.
@PermuteDeclr
field / param / for-each
Renames type and identifier per permutation value. Scope-aware: field renames propagate class-wide; for-each variable renames are contained to the loop body.
@PermuteParam
method / constructor param
Expands one sentinel parameter into a full generated sequence. Automatically rewrites every call site where the sentinel appears as an argument — no annotation at the call site required.
@PermuteTypeParam
type parameter
Expands a sentinel type parameter into a sequence. <A> becomes <A, B, C>. Implicit when using T${j} naming convention.
@PermuteConst
field / local var
Replaces an initializer with the evaluated JEXL expression. @PermuteConst("${i}") int ARITY = 2; becomes int ARITY = 3; in Join3.
@PermuteValue
field / var / method
Superset of @PermuteConst. Replaces the RHS of any assignment statement at a given index position in a method body — not just initializers.
@PermuteReturn
method
Controls the return type per permutation value. Enables stateful builder chains where each .join() returns the next step type with an additional type parameter.
@PermuteMethod
method
Generates multiple overloads from one method using an inner loop variable. Leaf classes automatically drop the method when the range is empty — no compile error.
@PermuteExtends
class
Explicit control over the extends/implements clause. Implicit same-N expansion fires automatically for standard T${j} naming — use this only when inference doesn't apply.
@PermuteCase
method / constructor
Expands a switch statement by inserting new cases per inner-loop value, fully inlined. The seed case and default are preserved; new cases appear before default.
@PermuteStatements
method / constructor
Inserts generated statements at the first or last position in a method body. Works with or without an inner loop. Applied after @PermuteValue.
@PermuteImport
class (repeatable)
Adds a JEXL-evaluated import statement to each generated class. Essential when generated types reference classes whose import can't be derived from the template's own imports.
